# 需求规格说明书

## 1. 方案名称

校园活动管理与报名签到平台

## 2. 项目定位

本系统面向高校校园活动管理场景，服务于学生、社团组织者/活动负责人、各级审核管理员以及系统管理员，实现活动申请、材料提交、分级审核、活动招募、学生报名、签到管理、通知发布、新闻展示和结项管理等功能。

系统目标是将分散、线下或半人工处理的校园活动审批与组织流程平台化、规范化，提高活动管理效率，方便学生及时获取活动信息并参与活动，同时降低管理员在审核、通知、流转和统计上的工作压力。

## 3. 建设目标

1. 学生发起活动申请、提交材料、查询进度的一体化管理。
2. 管理员对活动申请材料的分级审核与逐级流转。
3. 活动招募发布、学生报名、材料提交与审核。
4. 活动签到与过程记录，便于活动管理与后续结项。
5. 新闻资讯、公开通知、系统推送等信息发布。
6. 基于“组织号”的自动审核人匹配与待办推送，提高流程自动化程度。

## 4. 用户角色

1. 学生用户：浏览新闻/通知、查看招募活动、报名、提交材料、查看审核结果、签到。
2. 活动申请学生/社团负责人：除学生权限外，发起活动申请与结项申请，查看审批流转。
3. 一级审核管理员：接收自动分配的申请，审核材料并给出通过/驳回/补充。
4. 二级及以上审核管理员：对通过的申请继续审批并推送至下一级。
5. 系统管理员：维护组织信息、审核链配置、用户与角色、新闻通知发布与系统运行管理。

## 5. 总体业务主线

1. 活动申请与审批：申请→读取组织号→匹配审核链→待办推送→分级审核流转→结束。
2. 招募与报名：发布招募→学生报名→材料审核→公布结果。
3. 执行与签到：生成签到方式（二维码/码/手动）→记录签到→统计导出。
4. 结项与归档：负责人提交结项材料→按既定流程审核→归档。
5. 信息发布：新闻、公开通知、系统推送与流程提醒。

## 6. 核心功能模块（10 个）

1. 用户登录与身份认证（角色识别、权限控制）
2. 首页新闻与通知展示（列表/详情/置顶/筛选）
3. 活动申请与材料提交（草稿、提交、补交、记录查询）
4. 自动审核人匹配与流程流转（组织号→审核链→待办→节点状态）
5. 管理员审核管理（待办箱、历史、意见、通过/驳回/退回补充）
6. 活动招募发布与报名（招募信息、报名材料、报名审核）
7. 活动执行与签到（二维码/签到码/手动、防重复、统计导出）
8. 结项申请与结果查询（结项材料、进度、结果）
9. 公开通知与消息推送（全体/定向、已读未读、流程提醒）
10. 组织与审核链配置（组织号、审核链、启停、异常匹配处理）

## 7. 核心流程（摘要）

### 7.1 活动申请审批流程

登录 → 填表/上传 → 提交 → 组织号匹配审核链 → 一级审核 → 推送下一级 → 结束（通过/驳回/补充）

### 7.2 招募报名流程

发布招募 → 学生报名/提交材料 → 管理员审核 → 通知结果 → 生成参与名单

### 7.3 签到流程

发起签到 → 生成二维码/签到码 → 学生签到 → 记录与防重复 → 统计报表

### 7.4 结项流程

提交结项材料 → 按既定流程审核 → 更新状态 → 归档与查询

## 8. 数据对象（实体）

1. 用户（User）
2. 组织（Organization）
3. 活动申请（ActivityApplication）
4. 申请材料（ApplicationAttachment）
5. 审核记录（ApprovalRecord）
6. 招募活动（Recruitment）
7. 报名记录（Signup）
8. 签到记录（Checkin）
9. 新闻通知（Announcement）

详细数据库设计见：`docs/database-design.md`。

## 9. 权限划分（摘要）

- 学生：浏览信息、申请/报名/签到/结项、查询进度与结果。
- 审核管理员：审核申请与报名、处理待办、查看统计。
- 系统管理员：组织号与审核链配置、用户与角色管理、系统参数与异常处理。

## 10. 非功能需求

### 10.1 性能

- 常规查询 3 秒内响应（新闻/通知/列表/进度等）
- 支持多管理员并发处理待办
- 支持报名高峰的集中访问

### 10.2 安全

- 角色/权限隔离
- 材料与审核意见的访问控制
- 关键操作留痕（审计日志）

### 10.3 可用性

- 表单校验、错误提示明确
- 支持草稿与补交流程

### 10.4 可靠性

- 流程状态可追踪、可恢复
- 上传材料可追溯

### 10.5 可维护与扩展

- 模块相对独立，支持新增活动类型/新增审核层级
- 预留短信/邮件/微信等通知通道
